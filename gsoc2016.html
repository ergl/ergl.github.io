Lasp and the Google Summer of Code

2016-08-16

Over the past few months, I've been working in Lasp, a new programming model for distributed dataflow programming, as part of the 2016 Google Summer of Code. Here I want to share some of my experiences, as well as give an overwiev on what I worked on during that time.

My work was mostly focused on run-time optimizations, in particular, on removing intermediate values from computations.

When I started the summer, I had several vague ideas about how to tackle the project. However, after some research, and a fruitful meeting with one of my mentors, we arrived at a promising solution, combining graph theory and control flow analysis, called dynamic path contraction.

I spent most of my time adding ways to monitor the run-time while Lasp programs execute (we even added a control flow visualization tool, that would let you see how different functions and values would interact as the program ran). I also had to modify some internal libraries that we depended on to support our optimizations.

The list with all the changes I made to the main repository is here, the list of changes to the supporting library, gen_flow, can be found here.

After implementing dynamic path contraction in Lasp, we ran several benchmarks and found a 25% to 37% decrease on end-to-end distribution latency for value.

While the results are very promising, there are still some things that could improve them even more. In particular, there are certain types of intermediate values that the current implementation isn't able to remove, mainly those that involve binary functions. There are also some operations that aren't currently monitored, like stream, that introduces non-determinism into the system.

With that said, I look forward to keep working on these issues and be able to improve Lasp.

All in all, it's been an amazing experience. I've received a lot of support from my mentors and teammates. Thank you.
